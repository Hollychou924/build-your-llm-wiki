# 文章归档决策树

收到一篇文章/文档链接后，按以下顺序判断。

## A. 这篇文章是否要归档？
- 是用户主动发来的链接 → **先检查是否已存在同源条目**
- 若**不存在同源条目** → **直接归档**
- 若**已存在同源条目** → **不重复新建 raw**，改做“同源复核”：
  - 补文章评级（若还没有）
  - 检查是否有新增事实需要回写旧条目
  - 在 `daily/` / `memory/` 中记录复核
- 不是用户主动发来的，只是路过看到 → 看价值再决定，不自动入库

## B. 是否必须做基础留痕？
- 只要决定归档，就**必须**完成：
  1. `raw/YYYY-MM/YYYYMMDD-keyword.md`（若非同源复核）
  2. `daily/YYYY-MM-DD.md`
  3. `memory/YYYY-MM-DD.md`（workspace 本地备份）
- 这三项是底盘，不允许跳

## B2. 是否先做文章评级？
- **是，默认必须先做**
- 至少给出：类型、干货指数、知识增量、表达价值、行动价值、总评、一句话判断
- 评级用于驱动后续知识加工，不只是装饰

## C. 是否更新 `wiki/entities/`？
- 出现**新实体**（公司 / 产品 / 人物 / 赛事 / 模型 / 协议）→ **更新**
- 老实体出现**重大新进展**（新版本 / 排名跃迁 / 能力升级 / 合作融资）→ **更新**
- 否则 → **不更**

## D. 是否更新 `wiki/concepts/`？
- 出现**新概念 / 新范式 / 新方法论**，以后大概率反复出现 → **更新**
- 对已有概念给出更清晰定义、结构、trade-off → **更新**
- 否则 → **不更**

## E. 是否更新 `summaries/`？
- 同主题累计 3 篇以上，值得做阶段摘要 → **更新**
- 单篇信息密度极高，已经足以成为主题入口 → **更新**
- 否则 → **不更**

## F. 是否更新 `wiki/comparisons/`？
- 文章天然就是多对象对比，或后续大概率会被问“它和谁比怎么样” → **更新**
- 没有明确对照对象 → **不更**

## G. 是否更新 `wiki/syntheses/`？
- 已经能跨 2-3 篇文章形成更高层判断 → **更新**
- 还只是单篇事实，不足以上升到趋势 → **不更**

## H. 是否更新 `insights/`？
- 对用户当前项目有明确启发，能沉淀成一句值得反复引用的判断 → **更新**
- 只是“有意思”但没有稳定判断 → **不更**

## I. 是否更新 `index.md`？
- 只要 `wiki/`、`summaries/`、`insights/` 有新增或改动 → **必须更新**
- 如果只有 `raw/`、`daily/`、`memory/` 有变化 → **不用更新**

## 特殊说明：公众号文章 vs 普通网页

- **微信公众号文章**：
  1. **首选** `wechat-article-extractor-skill`
  2. **fallback**：curl + python 本地解析（仅在 skill 失败时使用）
- **普通网页文章**：
  1. **首选** `web_fetch`（OpenClaw 内置）
  2. 若 `web_fetch` 正文为空、或被隐藏，fallback 到 curl + python / readability 解析
